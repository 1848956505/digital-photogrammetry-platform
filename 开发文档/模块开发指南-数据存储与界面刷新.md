# 模块开发指南 - 数据存储与界面刷新

本指南说明插件模块如何与 `Workspace`、`EventBus` 和主窗口协作，目标是让所有模块以统一方式读写数据、保存结果、刷新界面。

## 1. 适用范围

本指南适用于所有 `plugins/mod*` 模块，尤其是图像处理、点云处理、矢量处理和成果导出类功能。

## 2. 当前实现约束

### 2.1 Workspace 访问方式
- 模块中必须通过 `get_workspace()` 获取工作空间单例。
- 不允许在插件中自行 `Workspace()` 新建实例。
- 模块应默认从 `self.workspace` 读取输入数据，而不是自行扫描磁盘。

### 2.2 资源池统一口径
当前版本统一采用以下资源池：
- `images`：原始影像
- `processed_images`：所有图像类处理结果
- `point_clouds`：点云
- `vectors`：矢量
- `masks`：掩膜
- `dom` / `dem`：项目级成果引用位

### 2.3 图像结果写回规则
所有图像类算法结果必须统一走以下路径：
1. 处理得到 `numpy.ndarray`
2. 调用 `self.workspace.add_processed_image(name, image_array)`
3. 通过 `EventBus` 发布 `TOPIC_IMAGE_UPDATED`
4. 由主窗口和 ImageViewer 自动刷新

推荐封装为统一模板：

```python
from datetime import datetime
from core.event_bus import EventTopics, get_event_bus


def _update_workspace_image(self, image, process_type: str):
    timestamp = datetime.now().strftime("%H%M%S")
    result_name = f"{process_type}_{timestamp}"
    self.workspace.add_processed_image(result_name, image)
    get_event_bus().publish(EventTopics.TOPIC_IMAGE_UPDATED, {
        "name": result_name,
        "path": self.workspace.get_processed_image(result_name)
    })
    return result_name
```

### 2.4 DOM / DEM 语义引用
- `processed_images` 是第一版的统一结果池。
- `dom` / `dem` 只作为项目级引用位，不替代 `processed_images`。
- 如果模块四或模块三需要标记“当前成果”，可以额外设置引用位，但仍应保留完整结果在 `processed_images`。

## 3. 插件目录规范

每个模块至少包含：
- `__init__.py`
- `plugin.py`
- `ui.py`

可选目录：
- `algorithms/`
- `models/`
- `docs/`

## 4. IPlugin 执行规范

### 4.1 当前实现约束
- `plugin_info()` 返回菜单信息、分组名、版本、描述等元数据。
- `get_ui_panel()` 返回右侧工具面板。
- `execute()` 负责流程编排。

### 4.2 推荐输入输出约定
- 输入默认从 `self.workspace` 读取。
- 输出默认写回 `processed_images` 或相应资源池。
- `execute()` 建议返回结构化状态：

```python
{"success": True, "message": "完成", "result_name": "XXX_123456"}
```

如果出现错误：
```python
{"success": False, "message": "失败原因"}
```

### 4.3 未来扩展预留
- 后续可以统一插件执行返回模型，但第一版不要让每个模块自行定义不同格式。

## 5. 界面刷新规范

### 5.1 当前实现约束
- 图像新增后发布 `TOPIC_IMAGE_ADDED`
- 图像被选中后发布 `TOPIC_IMAGE_SELECTED`
- 图像处理完成后发布 `TOPIC_IMAGE_UPDATED`

主窗口职责：
- 监听事件总线
- 刷新左侧资源树
- 必要时更新中央视图和状态栏

ImageViewer 职责：
- 监听 `TOPIC_IMAGE_SELECTED`
- 监听 `TOPIC_IMAGE_UPDATED`
- 自动重新加载并显示最新结果

### 5.2 资源树映射规则

| Workspace 字段 | 资源树节点 | 说明 |
|---|---|---|
| `images` | 原始影像 | 导入的源图像 |
| `processed_images` | 处理结果 | 灰度、滤波、重采样、DOM 临时结果等 |
| `point_clouds` | 点云 | 三维点云资源 |
| `vectors` | 矢量 | 线、面、边界等矢量数据 |
| `masks` | 掩膜 | 分割或遮罩结果 |
| `dom` / `dem` | 项目成果引用 | 语义引用位 |

### 5.3 未来扩展预留
- 可增加更细粒度的事件，例如 `TOPIC_DOM_UPDATED`、`TOPIC_POINTCLOUD_UPDATED`
- 可增加结果预览缩略图刷新机制

## 6. 命名规范

### 6.1 当前实现约束
处理结果建议命名为：

```text
处理类型_具体参数_时间戳
```

示例：
- `灰度变换_线性变换_103045`
- `滤波_均值滤波_103102`
- `重采样_50%_103115`
- `DOM_羽化拼接_144500`

### 6.2 命名原则
- 结果名要能看出来源模块和处理方式
- 名称尽量短，但要足够可读
- 如果同一模块会生成多种结果，参数应尽量写入名称

## 7. 保存与导出

### 7.1 当前实现约束
- 不要直接用 `cv2.imwrite()` 处理中文路径
- 建议使用 `cv2.imencode(...).tofile(...)` 保存文件
- 导出完成后若属于图像结果，应继续写回 `processed_images`

### 7.2 未来扩展预留
- DOM / DEM 可增加 GeoTIFF、JPEG2000 等导出分支
- 点云与矢量可增加专用导出器

## 8. 模块编写模板

一个标准插件模块建议包含以下逻辑：
1. `plugin_info()` 返回元数据
2. `get_ui_panel()` 创建 UI
3. UI 按钮连接到处理函数
4. 处理函数读取 workspace 数据
5. 算法处理完成后写回 workspace
6. 通过 EventBus 触发刷新
7. 返回状态对象或布尔值

## 9. 测试建议

### 9.1 单模块测试
- 输入为空时是否给出友好提示
- 处理参数变化时是否得到不同结果
- 输出是否正确写入 `processed_images`
- 是否正常触发 `TOPIC_IMAGE_UPDATED`

### 9.2 集成测试
- 处理结果是否出现在资源树中
- 中央视图是否自动刷新
- 多个模块连续执行时是否不会覆盖其他资源池

### 9.3 回归测试
- 不影响插件扫描
- 不影响项目保存/加载
- 不影响已有模块一功能

## 10. 开发红线

以下做法视为不符合规范：
- 在插件中直接创建新的 Workspace 实例
- 绕过 EventBus 直接改 UI
- 将图像处理结果只保存在内存变量中而不回写 workspace
- 让不同模块各自定义不同的结果命名风格
- 把 DOM / DEM 当成唯一结果池而忽略 `processed_images`

## 11. 结语

本指南的核心目标只有一个：让所有模块都遵守同一套读写和刷新规则。只要按这里的模板写，后续模块之间的协作成本会小很多。
